# Figure Extraction from PDF Specs — Brainstorm

**Status:** brainstorm. Explores needs and possibilities for
extracting figure content from PDF source specs. No code,
architecture, or implementation plan in this doc — those land
separately once we settle on a direction.
**Date:** 2026-05-17.
**Related:** [lancedb-rig-incorporation-plan.md](lancedb-rig-incorporation-plan.md),
[spec-md-restructure.md](spec-md-restructure.md),
the four-spec review thread in conversation.

## 1. Why figure extraction matters

Hardware specs depend on figures more than any other domain doc
we have looked at. The four real specs (Apical NoC, Numenta SoC,
Spatial Pooler, RV12) all carry critical information **only in
figures**:

- RV12 IF block diagram (p13) defines the next-PC mux: which
  signals (`pc+2`, `pc+4`, `ex_nxt_pc`, `st_nxt_pc`) feed which
  destination (`if_nxt_pc`), under what control (`Flush`/
  `Stall`). The prose says "the PC is restarted from the given
  address." The mux topology lives ONLY in the figure.
- RV12 EX bypass network (p20) defines the entire operand-
  forwarding fabric (`wb_r`/`mem_r`/`ex_r` into each execution
  unit). None of the bypass topology appears in prose.
- Apical NoC mesh topology figures (pages 1, 4, 6) carry the
  physical layout (which engines own which edges, which channel
  types connect which neighbours).
- Numenta SoC IO System diagram (p7) shows AXI vs APB widths
  per peripheral, an architectural fact missing from the prose.

A cycle-accurate model authored from prose alone will silently
omit the mux topology, the forwarding network, the mesh edges.
"Figures are decorative" is false for hardware specs.

## 2. What's actually happening in the current pipeline

`spec_ingest.rs` uses `pdfium-render` to paginate a PDF into
per-page markdown and to extract embedded raster images into
`.sim-flow/spec-pages/images/`. On RV12 (95 pages) this produces
14 extracted images. We hand-reviewed all 14:

**Render correctly** (text labels and connecting structure
intact):

- `page-002` — top-level RV12 block diagram (all 14 labeled
  boxes preserved).
- `page-006`, `page-007-img-18` — pipeline-stage box sequences
  (text rendered rotated 90°; readable).
- `page-007-img-19` — instructions-in-flight timing chart.
- `page-012` — full execution pipeline with ICACHE / Bus
  Interface / exec units.

**Useless** (shapes only — text labels gone, connecting wires
gone):

- `page-013` — IF block diagram. Shapes (mux trapezoids, FF
  rectangles, cloud "logic" blobs) present; every signal label
  and every wire is missing.
- `page-015` — PD internals. Same shape-only result.
- `page-016`, `page-017` — ID internals. Same.
- `page-020` — EX block diagram. Block names
  (`Load Store`/`Branch`/`Div`/`Mul`/`ALU`/`OpA`/`OpB`) survive
  because they live inside the embedded raster; every forwarding
  signal label and the entire mux network are missing.
- `page-022`, `page-024` — empty vertical columns of rectangles.

**Noise**:

- `page-001` — a 241-byte blue rectangle (the "View on GitHub"
  button background).

## 3. Why this happens

This is a structural property of how the source PDF was authored,
not a tuning issue. Hardware datasheet figures fall into two
classes:

### Class A — Self-contained vector figures

The whole figure is drawn with PDF vector ops: rectangles, lines,
text, fills, all in the page's content stream. When the figure
is "extracted" by rasterizing the bounding box (or sometimes
when it's grouped as a Form XObject), labels and structure ride
along. **RV12's top-level block diagram (p2) is this class.**
Apical NoC's mesh layouts are this class.

### Class B — Composite figures (raster sheet + vector overlay)

The figure is composed of:

1. **An embedded raster image** (PNG/JPEG inside the PDF as an
   Image XObject) containing only the rectangular shapes,
   trapezoid muxes, and cloud "logic" blobs.
2. **PDF vector text-rendering ops** painted over the raster at
   absolute coordinates, supplying the signal labels
   (`if_nxt_pc`, `pc+2`, etc.).
3. **PDF vector path ops** painted over the raster, supplying
   the connecting wires between shapes.

A human reader (or a PDF renderer) sees the composite. The
"extract embedded images" API only returns (1) — the shape
sheet. Labels (2) and wires (3) are dropped on the floor
because they're not part of any image object. **Every RV12
per-stage diagram (p13, p15, p16, p17, p20) is this class.**
Likely most figures generated by tools like Visio, Lucidchart,
draw.io, or PowerPoint and exported to PDF.

## 4. What downstream consumers need from figures

Driving requirements rather than solution choices. Each consumer
has different fidelity needs:

### 4a. The agent during DM0 / DM1 (reading the spec)

Needs **a faithful visual of each figure** AND ideally a
**structured caption** describing what's depicted. The visual is
to confirm that the structured caption matches reality (for
review by the human or by a critique pass). The caption is what
gets text-embedded for retrieval and threaded into prompts.

### 4b. The agent during DM2 / DM3 (writing the model)

Needs the **structured caption** — element list, connection
list, control signals. Doesn't need the raw pixels at this
stage. If the caption is wrong, the model will be wrong; the
visual fidelity matters only insofar as it lets DM1 author a
correct caption.

### 4c. The critique pass (DM*-critique)

Needs both: the visual to verify the caption, and the caption
to verify the model. Likely the highest-leverage place for a
vision-model spot-check pass.

### 4d. The human reviewer

Needs the visual (rendered at presentation quality) plus
references in spec.md back to the source page.

### 4e. The L2 spec RAG index (future)

Needs the caption text indexed alongside the surrounding prose
chunk. The figure's caption becomes searchable as text — "where
does `ex_nxt_pc` come from?" hits the IF figure's caption.

## 5. Solution space (possibilities, not decisions)

Five candidate approaches, scored for fidelity (does the result
match what a human sees?) and effort.

### S1. Page-region rasterization

Instead of "extract embedded images," render the full PDF page
to a raster at sufficient DPI and crop to figure regions. The
rendered raster is the composite — shape sheet + text overlay +
wire overlay — exactly what a human sees.

- **Fidelity:** high. Matches the printed PDF.
- **Effort:** small. pdfium-render exposes
  `render_with_config` → `PdfBitmap`. Page → PNG is a known
  recipe; cropping requires detecting figure bounding boxes
  (heuristic, or a second pass).
- **Trade-off:** larger on-disk footprint (a 95-page spec at 150
  DPI is tens of MB vs the current 928 KB). For RAG / vision-
  model consumption that's fine; for storage it's not free.
  Mitigation: render JPEG, only for pages that contain figures.

### S2. Hybrid: extract embedded image, fall back to page render

Keep the current extract-image path for Class A figures (already
working). When the extracted image fails a sanity check ("does
the raster contain text?"), render the page region instead.

- **Fidelity:** high where it matters.
- **Effort:** medium. Needs a heuristic to detect Class A vs B,
  or a "no text in image, but text near image in PDF stream"
  signal.
- **Trade-off:** more code paths to maintain. Worth it only if
  the page-render storage cost matters.

### S3. Page render + lightweight figure cropping

Render every page that contains any drawing op, save the whole
page as one image, and let downstream consumers crop based on
the page's bounding-box hints from pdfium. Avoids per-figure
cropping logic entirely.

- **Fidelity:** high.
- **Effort:** small.
- **Trade-off:** consumers see whole pages, not figures. May
  bloat vision-model prompts; may be fine for direct human
  reference. Downstream tools can do their own cropping.

### S4. Vector-extraction (reconstruct the figure from PDF ops)

Parse the PDF content stream directly to recover text positions,
path operations, and image references, then re-render to a
clean raster (or to an SVG) with everything composited. Most
faithful possible reconstruction.

- **Fidelity:** highest.
- **Effort:** high. Effectively writing a PDF renderer subset.
  pdfium can render but doesn't expose primitive-level
  semantics easily.
- **Trade-off:** overkill if S1 / S2 produce visually identical
  output via pdfium's existing renderer.

### S5. Outsource to a vision-PDF tool

Hand pages to a tool like `pdfplumber`, `unstructured`, Mathpix,
or a hosted document-AI API that does figure extraction +
captioning end-to-end.

- **Fidelity:** variable (depends on the tool).
- **Effort:** small to integrate, ongoing dependency / cost.
- **Trade-off:** introduces another transport (HTTP API or
  Python tool); pulls us off the embedded-Rust path. Worth
  evaluating if S1–S4 don't pan out, but should not be the
  default.

## 6. The rotation problem

Separate from extraction fidelity. Several RV12 figures render
text rotated 90° (so labels are read bottom-to-top or
top-to-bottom). This is fine for human readers but problematic
for OCR or vision-model captioning if not normalized.

Options worth considering:

- **Detect at extract time** (text annotations in the PDF
  content stream carry their text matrix; we can read the
  rotation angle).
- **Normalize in the rendered raster** (rotate the whole page
  or just the figure region before saving).
- **Tolerate** — modern vision models (Claude vision, GPT-4o)
  handle rotated text reasonably; we don't need to pre-rotate
  for them.

Likely fine to defer until we've picked a captioning path.

## 7. Captioning: needs but no chosen path

Once we have faithful figure rasters, the next question is how
to turn them into structured captions that downstream consumers
can use. Options (brainstorm only):

- **Vision model at ingest time.** Send each figure raster +
  surrounding prose to a vision LLM, produce a structured
  caption (element list, connection list, control signals).
  Highest fidelity, costs tokens per figure per ingest.
- **Author-supplied captions.** Require the spec author to
  provide a markdown / TOML caption per figure. Zero compute
  cost, requires author effort, opt-in.
- **Hybrid: vision-model draft, author review.** Vision model
  produces a caption that the author edits / approves. The
  spec gets an `images/<name>.caption.md` sibling that the
  ingest path picks up.
- **DM0 turn captures captions interactively.** The agent in
  manual mode asks the user to describe each figure that
  appears in scope for the current step.
- **Defer captioning entirely; just surface the raster.** The
  agent / human reads the rendered figure on demand. Loses the
  retrievability benefit but is the smallest change.

What we want from a caption (regardless of who produces it):

- The **role** of the figure ("IF stage internal block diagram",
  "top-level execution pipeline").
- The **elements** depicted, with names matching the
  surrounding prose (signals, blocks, buses).
- The **connections** between elements (directional edges,
  optional labels).
- The **controls** (clocks, flushes, stalls).
- A **back-reference** to the source spec page.

## 8. Open questions

1. **DPI and storage budget.** What page-render DPI gives
   readable text on hardware figures without bloating disk?
   150 DPI handles RV12 labels but is untested on tighter
   layouts (Numenta SoC, Apical NoC tables-within-figures).
2. **JPEG vs PNG.** PNG is lossless but bigger; JPEG is smaller
   but degrades text. JPEG at quality ≥85 should be fine for
   figures with prose-sized labels; verify on the four sample
   specs.
3. **Figure bounding-box detection.** Worth doing inside the
   ingest, or punted to downstream consumers? S3 vs S1 hangs
   on this.
4. **What about non-figure embedded images** (logos, "View on
   GitHub" buttons, decorative elements)? Heuristic filter on
   size / aspect ratio? Or just let them through and let the
   agent ignore?
5. **Cross-page figures.** Apical NoC and Numenta SoC have
   figures embedded mid-prose without dedicated page numbers.
   Do bounding boxes survive across paragraphs?
6. **PDFs that are themselves OCR'd / scanned.** None of the
   four sample specs is in this category, but if the project
   admits scanned-spec inputs eventually, S1 still works (the
   "page" is one big raster) but text overlays don't exist —
   different failure mode.
7. **Where do captions live?** Sibling files
   (`<image>.caption.md`)? Inline in the page markdown?
   Separate manifest (`figures.toml`)? Storage shape affects
   how the spec.md restructure (see
   [spec-md-restructure.md](spec-md-restructure.md)) references
   figures.
8. **Caption schema.** TOML, JSON, structured markdown? Should
   align with the signal-table extraction schema in L0f / L7
   so that "elements depicted in figure X" can join against
   "signals in stage X."

## 10. Bottom line (brainstorm-level)

Two findings from looking at the actual extracted images:

1. **The current pipeline silently loses critical information**
   on a known, identifiable class of figures (PDF composite
   figures — raster sheet + vector text/wire overlay). RV12's
   IF and EX block diagrams are the canonical failure case.
   This is a correctness regression we should fix regardless
   of the lance/rig direction.

2. **The fix is well-bounded.** Page-region rasterization
   (S1 or S3) produces visually faithful figures using
   pdfium's existing renderer. This is a pdfium API change in
   `spec_ingest.rs`, not a new dependency.

Captioning (turning a faithful figure into structured text the
agent can use) is the second-half problem and has more options
to explore. The first-half problem (getting a faithful figure
out of the PDF) has a clear path.
